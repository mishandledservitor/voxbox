# Agent guide — voxbox

This file is for AI agents (and humans) onboarding to the repo. It covers layout, setup, and the commit/push workflow across the parent repo and its submodules.

## Repo layout

`voxbox` is a parent repo containing five submodules, each its own GitHub repo under `mishandledservitor`:

| Path              | Remote                                                | Purpose                                  |
|-------------------|-------------------------------------------------------|------------------------------------------|
| `kokoro-tts/`     | `mishandledservitor/kokoro-tts`                       | Local TTS engine                         |
| `whisper-stt/`    | `mishandledservitor/whisper-stt`                      | Local STT (Whisper)                      |
| `whisper-diarize/`| `mishandledservitor/whisper-diarize`                  | STT + speaker diarization (optional)     |
| `speech-to-text/` | `mishandledservitor/speech-to-text`                   | Higher-level STT pipeline / transcripts  |
| `assemblyai-stt/` | `mishandledservitor/assemblyai-stt`                   | AssemblyAI cloud STT wrapper             |

Top-level Python entry points: `voxbox_cli.py` (CLI), `voxbox_gui.py` (Tk GUI), and the `voxbox` launcher script generated by `setup_voxbox.sh`.

## First-time clone

```bash
git clone --recurse-submodules https://github.com/mishandledservitor/voxbox.git
cd voxbox
./setup_voxbox.sh
```

If you forgot `--recurse-submodules`:

```bash
git submodule update --init --recursive
```

`setup_voxbox.sh` initializes submodules, runs `kokoro-tts/setup_kokoro.sh` and `whisper-stt/setup_whisper.sh`, optionally runs `whisper-diarize/setup_whisper_diarize.sh`, and writes the `./voxbox` launcher.

`assemblyai-stt/` has its own `setup_assemblyai.sh` and needs an `ASSEMBLYAI_API_KEY` in `assemblyai-stt/.env` (gitignored).

## Pulling updates

```bash
git pull
git submodule update --init --recursive
```

To advance a submodule pointer to the tip of its remote `main`:

```bash
git submodule update --remote --merge <path>
```

## Commit + push workflow (agent checklist)

The parent repo tracks submodules by **gitlink** (a pinned SHA). When a submodule changes, you commit + push **inside the submodule first**, then commit the updated pointer in the parent. Submodules are usually on detached HEAD after `git submodule update` — check out `main` before committing.

```bash
git status            # see which submodules are dirty
git submodule status  # verify pinned SHAs and branches
```

For each dirty submodule:

```bash
cd <submodule>
git checkout main && git pull --ff-only
# stage and commit work
git add -A
git commit -m "<message>"
git push origin main
cd ..
```

Then in the parent:

```bash
git add <submodule>           # stages the new gitlink SHA
git add <other parent files>  # e.g. .gitmodules, voxbox_*.py
git commit -m "<message>"
git push origin main
```

Verify clean:

```bash
git status                                   # both parent and any submodule should be clean
git submodule status                         # SHAs match what's on origin/main
git log origin/main..HEAD                    # empty in parent + each pushed submodule
```

## Adding a new submodule

```bash
# In the new project directory (no .git yet):
cd new-thing/
# Add a .gitignore that excludes .env, venv/, __pycache__/, and any data dirs BEFORE git init
git init -b main
git add -A && git commit -m "Initial commit"

# Create the GitHub repo under mishandledservitor (private by default).
# This requires gh auth as a user with mishandledservitor access:
gh repo create mishandledservitor/new-thing --private --source=. --remote=origin --push

# Then register it as a submodule of voxbox:
cd /path/to/voxbox
rm -rf new-thing                             # remove the inline working copy
git submodule add https://github.com/mishandledservitor/new-thing.git new-thing
git commit -m "Add new-thing submodule"
git push origin main
```

## Important rules for agents

- **Never commit secrets.** Always check for `.env` files and API keys before `git add -A` in a fresh directory. The `assemblyai-stt/.env` contains a real API key — it is gitignored; keep it that way. The same applies to any new submodule.
- **GitHub auth.** Submodules live under the `mishandledservitor` org. Confirm the active `gh` account can write there before running `gh repo create` — `gh auth status`. Do not fall back to a different org without explicit user approval.
- **Submodule branch hygiene.** `git submodule update` leaves submodules on detached HEAD. Always `git checkout main && git pull --ff-only` before committing inside a submodule, otherwise your commit won't be on `main` and the push will fail or be lost.
- **Pointer + content together.** A submodule content commit and the parent's gitlink bump are two separate commits in two separate repos. Push the submodule first; otherwise the parent points at a SHA that doesn't exist on the remote.
- **No force-push, no `-A` at the repo root without reading `git status` first.** This repo includes data directories (`inbox/`, `output/`, `processed/`, `transcripts/`) — review what's being staged.
